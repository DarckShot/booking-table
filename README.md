# SAVEUR — бронирование столика

Тестовое задание для позиции **Junior Frontend Developer / React · Next.js**.

Проект представляет собой страницу онлайн-бронирования столика в ресторане SAVEUR. Реализована форма бронирования, клиентская валидация, имитация отправки заявки, экран подтверждения и адаптивная вёрстка для мобильного и десктопного разрешений.

## Что реализовано

В проекте реализована форма бронирования столика с полями: имя гостя, телефон, дата, время и количество гостей. Все поля обязательны для заполнения. Для даты используется кастомный календарь, для времени — кастомный выпадающий список со слотами с 12:00 до 22:00. Также реализована кастомная кнопка с состоянием загрузки.

Валидация работает при потере фокуса и при попытке отправить форму. Ошибки отображаются под соответствующими полями, а поля с ошибками получают красную обводку. Форма не отправляется, если есть ошибки валидации.

Отправка заявки имитируется через задержку в 1.5 секунды. Во время отправки кнопка блокируется и показывает состояние `Бронирую...`. После успешной отправки отображается экран подтверждения с деталями бронирования и кнопкой `Забронировать ещё`, которая возвращает пользователя к форме.

Также добавлены базовые анимации перехода между формой и экраном подтверждения, базовая настройка доступности для интерактивных элементов и unit-тесты для логики валидации, работы с датами, календарных утилит и пользовательских хуков.

## Правила валидации

**Имя гостя** — обязательное поле. Минимум 2 символа. Разрешены только буквы, пробелы и дефис.

**Телефон** — обязательное поле. Поддерживается российский формат номера: `+7` или `8`, далее 10 цифр. При вводе можно использовать пробелы, скобки и дефисы — перед проверкой номер нормализуется.

**Дата** — обязательное поле. Нельзя выбрать дату раньше сегодняшнего дня и позднее чем через 90 дней.

**Время** — обязательное поле. Доступны только фиксированные слоты с 12:00 до 22:00 с шагом 1 час.

**Количество гостей** — обязательное поле. Допустимо только целое число от 1 до 12.

## Тесты

В проекте используются `Vitest` и `happy-dom`. `happy-dom` добавлен для тестирования пользовательских хуков, которым требуется окружение браузера: `document`, обработчики событий, закрытие выпадающих элементов по клику вне компонента и по клавише `Escape`.

Тестами покрыты:

- правила валидации формы;
- утилиты для работы с датами;
- логика генерации календарной сетки;
- ограничения дат по `min` и `max`;
- состояние выбранной даты и текущего дня;
- пользовательские хуки кастомных элементов управления.

## Доступность

Для полей и кастомных элементов управления добавлены базовые атрибуты доступности. Ошибки связаны с полями через `aria-describedby`, состояние ошибки передаётся через `aria-invalid`. Для выпадающего списка используются `aria-expanded`, `aria-haspopup`, `role="listbox"` и `role="option"`. Кастомные декоративные иконки скрыты от скринридеров через `aria-hidden`.

Также предусмотрено закрытие кастомного календаря и выпадающего списка по клавише `Escape` и по клику вне компонента.

## Принятые решения

Для работы с формой используется `react-hook-form`, так как он удобно обрабатывает валидацию, состояние отправки и ошибки полей без лишнего усложнения кода. Правила бронирования вынесены в отдельные константы, типы и утилиты, чтобы компоненты оставались читаемыми. Календарь и выпадающий список сделаны кастомными, потому что так проще сохранить единый визуальный стиль формы. Кнопка вынесена в отдельный переиспользуемый компонент, так как у неё есть общее поведение: варианты отображения, состояние загрузки и блокировка при отправке. Анимация между формой и экраном подтверждения реализована через Framer Motion, чтобы переход выглядел мягко, но не перегружал интерфейс.

## Что можно было бы улучшить

При наличии дополнительного времени я бы доработал полноценную клавиатурную навигацию внутри кастомного календаря и выпадающего списка. Также можно было бы улучшить управление фокусом при открытии и закрытии интерактивных элементов. Для телефона можно добавить маску ввода, чтобы пользователю было удобнее вводить номер. Ещё можно расширить тесты и проверить полный сценарий бронирования через React Testing Library.

## Примечания

В проекте нет backend-части, базы данных, авторизации и отдельных страниц, так как это не требовалось по условиям тестового задания. Отправка формы имитируется на клиенте через задержку в 1.5 секунды.
